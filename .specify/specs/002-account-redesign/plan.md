# Implementation Plan: Account Page Redesign & User Avatars

**Branch**: `002-account-redesign` | **Date**: 2026-06-02 | **Spec**: `.specify/specs/002-account-redesign/spec.md`

## Summary

Complete rewrite of `Account.vue` using a three-zone layout (profile hero → security →
danger zone), a new `AvatarPicker.vue` component offering initials / civilization-crest /
custom-upload sources, and a `useAvatar.js` composable that resolves the stored reference
to a display URL. Avatar choice is persisted to Firestore `users/{uid}` and reflected live
in the profile hero and app header. Custom uploads are resized client-side to 256×256 WebP
before uploading to Firebase Storage, keeping per-user image cost negligible.

## Technical Context

**Language/Version**: JavaScript (ES2022), Vue 3.2 SFCs

**Primary Dependencies**: Vue 3, Vuetify 3, Vuex 4, Firebase Auth, Firebase Firestore,
Firebase Storage (already configured in project, not yet imported in `firebase/index.js`)

**Storage**:
- Civ / initials choice → `users/{uid}` Firestore document (1 write, ~30 bytes)
- Custom upload → Firebase Storage bucket at `avatars/{uid}.webp` (one file per user,
  max ≈30 KB after client-side resize)

**Testing**: Manual golden-path per Constitution. No automated test suite.

**Target Platform**: Web SPA (desktop + mobile), Netlify-hosted

**Project Type**: Vue 3 frontend, single `src/` project

**Performance Goals**: Avatar load is synchronous for initials and civ crests (bundled
assets); upload progress shown inline; no blocking reads at page load

**Constraints**:
- Vuetify components only; existing theme tokens (Constitution III)
- No new npm dependencies — canvas API is browser-native; Firebase Storage SDK is
  already in the project's `firebase` package (just not yet wired)
- Firebase Storage free tier: 5 GB storage, 1 GB/day download — well within hobby limits
- Firestore `users/{uid}` write on every avatar change (~1 write per user action)

**Scale/Scope**: 5 files changed/created, 1 existing view rewritten, 1 Firestore rule
extended, 1 Firebase Storage rule added.

### Key design decisions

- **Avatar composable** (`useAvatar`): takes `{ avatarType, avatarRef }` from Vuex store
  state; returns `{ src: string | null, initials: string }`. Template uses `v-avatar` with
  `<v-img>` when `src` is set, or the existing initials string when not.
- **Vuex avatar state**: a new `userAvatar` slice `{ type, ref }` loaded alongside
  `userFavorites` in the `onAuthStateChanged` handler. Cleared on logout.
- **AvatarPicker**: a `v-dialog` triggered by clicking the current avatar; three tabs via
  `v-tabs` — Initials (one click), Civilizations (grid of `flagSmall` images from
  `civDefaultProvider`), Upload (file input + canvas resize preview). Save dispatches
  `updateAvatar` action.
- **Upload resize**: browser `canvas` API — no library. Resize to 256×256, crop center,
  export as `image/webp` at quality 0.82. Same pattern used in `BuildImport.vue`.
- **Civ flag path**: reuse `civDefaultProvider.js` `flagSmall` values
  (`assets/flags/{code}-small.webp`) — already bundled, no Storage read needed for crests.
- **Header**: replace the existing `v-avatar` initials block with a call to `useAvatar()`;
  the template already has the right structure, just needs the `src` binding added.
