# Implementation Plan: Desktop Build View & Editor Layout

**Branch**: `014-desktop-build-layout` | **Date**: 2026-06-12 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `.specify/specs/014-desktop-build-layout/spec.md`

## Summary

Refresh the desktop (`md-and-up`) build hero and build-order table to match the mobile redesign bar: a civilization lockup in the header (flag + civ name, distinct from chips), a fixed five-column resource grid with calculated villager total, gold age-transition lane, increase-only delta accent, hover insert line (no "Add step" button), view/edit parity (SC-002), and WYSIWYG section note rows. The mobile layout (`011`) and all business logic are untouched.

The critical technical constraint is preserving the existing content-editable state management chain. The desktop `v-table` is kept (Constitution III) with column widths defined via `<colgroup>` and insert lines as `<tr class="ins-row">` siblings. See `research.md` for the preservation contract.

## Architectural Decisions

### A. Desktop table ÔÇö keep `v-table` with `<colgroup>` widths (Constitution III)

**Decision**: Keep the existing desktop `v-table` in `BuildOrderSectionEditor.vue`. Column widths are fixed using a `<colgroup>` / `<col width="ÔÇª">` block, eliminating auto-sizing drift without a JS pass. This is the Constitution-III-compliant choice over a custom CSS grid.

**Why not CSS grid**: CSS grid is custom HTML + CSS; `v-table` is the Vuetify-native component. The visual goals (fixed-width columns, alignment across sections) are achievable with `<colgroup>` column widths, which also makes `alignTableColumnWidthsAcrossSections()` redundant.

**Column widths** (adapted from `design-input.md ┬º3.1`):
```html
<colgroup>
  <col style="width:64px">   <!-- time -->
  <col style="width:44px">   <!-- villagers (read-only) -->
  <col style="width:52px">   <!-- builder -->
  <col style="width:52px">   <!-- food -->
  <col style="width:52px">   <!-- wood -->
  <col style="width:52px">   <!-- gold -->
  <col style="width:52px">   <!-- stone -->
  <col>                      <!-- description (fills remaining width) -->
  <!-- edit mode only: --> <col style="width:30px"> <!-- Ô£ò -->
</colgroup>
```
Columns: time ┬À villagers ┬À Builder ┬À Food ┬À Wood ┬À Gold ┬À Stone ┬À description (+ Ô£ò in edit).

### B. Content-editable DOM access ÔÇö minimal selector update (critical ÔÇö the user's explicit concern)

**Decision**: Keep `stepsTable` ref and `descriptionColumnIndex = 7`. Change only the row selector from `rows[index]` to `querySelectorAll('tr.step-row')[index]` so that insert rows (Decision C) don't corrupt the index.

**The invariant that must hold**: every call to `contentEditableHelper.js` functions (`updateSearchText`, `addAutocompleteIcon`, `addInlineIcon`), to `saveSelection`/`restoreSelection`, and to `handleIconSelectorIconSelected` receives a valid `contenteditable` DOM element. Only how the row is located changes ÔÇö the functions themselves are unchanged.

**Migration** (minimal):
- `stepsTable` ref stays, bound to `<tbody>` as before
- `descriptionColumnIndex = 7` stays unchanged
- Replace every `stepsTable.value.rows[index]` with `stepsTable.value.querySelectorAll('tr.step-row')[index]`
- Mark every data row: `<tr class="step-row" ÔÇª>` (add class; no other change)
- `noteRefs` and `registerNoteRef` are **not** added ÔÇö no migration needed
- `gameplanContentEditable` ref stays, bound to the gameplan `<td>` directly (unchanged)

See `research.md` for the complete preservation contract and all touch-points.

### C. Insert line ÔÇö `<tr class="ins-row">` in `<tbody>`

**Decision**: The hover insert line is a `<tr class="ins-row">` rendered between data rows in `<tbody>` (one before each step row in edit mode + a trailing one after the last step). Because insert rows are not `.step-row`, `querySelectorAll('tr.step-row')[index]` is unaffected by their presence.

**Why `<tr>` not `<div>`**: HTML `<table>` does not allow `<div>` siblings of `<tr>` inside `<tbody>`. A `<tr>` is the only valid in-flow element. A single-cell `<tr>` with `colspan="9"` and minimal height (4ÔÇô6 px) containing the gold insert affordance is clean and semantically valid.

**Index preservation**: Data rows carry class `step-row`; insert rows carry class `ins-row`. The querySelectorAll in Decision B targets only `.step-row` ÔÇö insert rows are invisible to it.

### D. `alignTableColumnWidthsAcrossSections` removal

**Decision**: Delete `alignTableColumnWidthsAcrossSections()` from `BuildOrderEditor.vue` and remove all its call-sites (`onMounted`, `nextTick` wrapper, `@textChanged` handler).

**Why**: Fixed `<colgroup>` widths applied identically in every section table enforce identical column positions automatically. The JS alignment pass only existed to compensate for HTML `<table>` auto-sizing variance. This is a Constitution I (Simplicity First) improvement.

**Risk**: None ÔÇö the function has no external callers and fixed `<colgroup>` widths make it redundant.

### E. BuildHeader.vue ÔÇö desktop civ lockup

**Decision**: In the desktop section (`d-none d-md-block`), add a flex row with the civ flag (`v-img`, 46├ù34, rounded) + civ name (`text-h6 font-weight-bold`) as the body column's leading element, before the chip group. Remove the civ chip from the chip group. The flag column (`.build-header__flag`) already exists and provides the full-bleed flag background ÔÇö the new lockup is an additional explicit civ-name text element in the body column.

**Why**: FR-001/002 require the civ to be prominently distinct from metadata chips. The existing desktop body shows title + chips where civ is just another chip. Adding flag-img + name text as a prominent lead element satisfies the lockup requirement without rebuilding the header structure.

### F. Age-up seeding + focus (FR-021)

**Decision**: Update `ageUp()` in `BuildOrderEditor.vue` to emit a `focusNewTimestamp` request after adding sections. Add a `timestampRefs` array in `BuildOrderSectionEditor.vue` ÔÇö each entry is a ref to the timestamp `<input>` (or `<td>`) for that step ÔÇö so the parent can focus it after `nextTick`.

The current `ageUp()` seeds `steps: [{}]` (one empty step per section) ÔÇö this is correct. What's missing is the focus step. After `await nextTick()`, the new section editor's `timestampRefs.value[0]?.focus()` is called. `timestampRefs` is separate from the `stepsTable` tbody ref and is unaffected by insert rows.

## Phase 0: Research

Research is complete and captured in `research.md`. All unknowns from the Technical Context above are resolved from direct codebase reading ÔÇö no external research required for this feature.

**Key findings** (details in `research.md`):
- `contentEditableHelper.js` is safe to use unchanged ÔÇö migration only affects *how* the contenteditable element is obtained, not how the helper functions work
- `document.execCommand("insertHTML")` is used for icon insertion ÔÇö deprecated but still universally supported; consistent with the `011` approach; do NOT replace it
- The "two copies" pattern (`steps` + `stepsCopy`) must be preserved for the description field; `querySelectorAll('tr.step-row')[i]` must resolve before structural mutations (addStep/removeStep)
- `alignTableColumnWidthsAcrossSections` can be safely deleted ÔÇö fixed `<colgroup>` widths make it obsolete
